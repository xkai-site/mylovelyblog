## 网关：Gateway

我们常做鉴权/限流/安全等非业务逻辑，当然最重要的还是路由分发。项目中的 application 会写以下配置：

```yaml
gateway:
  routes:
    - id: user-route
      uri: lb://service-user
      predicates:
        - Path=/api/users/**
      filters:
        - StripPrefix=1
```

主要看 predicates 和 filters，只要前端匹配上 users，那么转发到对应 uri 的 `lb://service-user`。

这个 LB 是负载均衡组件，它根据服务名去找 Nacos，Nacos 给它当前服务对应的多台机器，完成软件->硬件的映射，然后 LB 再根据这个映射选择一台机器转发。

![LB 负载均衡](/mylovelyblog/assets/images/微服务项目快速入门/Pasted image 20260407214045.png)

微服务中还有使用到 LB 的场景，就是服务调用。由于服务之间互不可知，只能通过 Nacos 进行获取，但 Nacos 本质上只是维护了一张注册表，并没有转发功能，因此仍然需要 LB。

我们只需要根据上述图片过一下链路既可，并不需要死记硬背。从第一性原理来看很简单：
- Nacos 是注册表，无法做转发功能。
- LB 是策略器，从多机器选一。
- OpenFeign/Gateway 做最后的调用。

## 开发工程点

### OpenFeign 与 Gateway

我们前面能明显感觉到它们之间对 LB 和 Nacos 的调用都是一致的，那为什么不将它们合并起来，减少逻辑负担呢。本质在于它们职责不同：
- Gateway 对外，需要提供鉴权/限流等等功能，这一套几乎是防御措施。
- OpenFeign 对内，内部服务的流动要求的是高效。

### Nacos 的用法

Nacos 既是注册中心，又是配置中心。

**配置中心：**
这个比较简单但又容易踩坑，总体来说两点：
- `@RefreshScope` 注解，能够实现配置热重载。这里是通过 Bean 重建实现的，因此需要落在注入配置的 Bean 上。
- 配置需要配在 Nacos 上才能真正实现配置与运行分离。项目中的 application 可能会提供配置，但只能在 Nacos 未生效时起作用，且无法实现配置与运行分离。

**注册中心：**
只需要在 application 中做配置，注意和 Sentinel 的配置地址做好区别，这样就完成了注册。
注册好后还要发现别人，因此我们要在模块启动类配置@EnableDiscoveryClient注解，这样就有了服务发现的能力。

### OpenFeign 的用法

OpenFeign 主要用于服务调用，因此注重共享。我们可以将需要跨服务调用的方法通过门面模式以内部 Client 暴露出来。值得一提的是，OpenFeign 也是可以做降级/兜底的。

### Sentinel 的用法

Sentinel 主要用于维护内部安全，提供降级/兜底策略。维护内部，我们可以用 `@SentinelResource(value = "createOrder", blockHandler = "createOrderBlocked", fallback = "createOrderFallback")` 直接实现正常/降级/兜底策略。当然也可以在 Sentinel 配置更简洁的规则策略。

一般来说：**Sentinel(降级) > OpenFeign(兜底) > Sentinel(兜底)**，这与执行顺序有关。

### Seata 的用法

Seata 需要 undo_log 表，这个表需要显示创建才能生效。在需要使用分布式回滚的情况下使用全局链，配合 undo_log 做回滚：
`@GlobalTransactional(name = "create-order-global-tx", rollbackFor = Exception.class)`

### Zipkin

这个主要方便链路记录，配合日志使用。在服务量少时它并不是必要的，但在服务增多的情况下显得非常可贵。后续计划配合 MQ 做文章。

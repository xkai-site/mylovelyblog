先看看调度器管理CPU的四种典型策略

| 策略        | 如何管理CPU                         |
| :---------- | :---------------------------------- |
| 全局队列    | 所有CPU共享一个队列，谁空闲谁取任务 |
| Per-CPU队列 | 每个CPU独立队列，任务固定在某CPU    |
| 推拉模型    | 空闲CPU主动去忙CPU拉任务            |
| 混合调度    | 实时任务固定CPU，普通任务共享队列   |

我现在想把Agent作为cpu用，很多情况下用户的并发量有限制，而缓存亲和也是重要的（能够更好的缓存命中减少成本）。
我还没看操作系统之前，用的是类似Per-CPU队列，这是我自然想到的。每个Agent根据skill注入，处理对应的任务比较简单。然后只做领域内的工作，也是缓存亲和的。
但在ReAct中，实际的工程实践需要将上下文重复发给Api，这样是不是相当于每次都要skill注入，所以cpu与队列对应的优势相当于大大削弱了（仅剩下上下文注入的速度）。此外大部分使用场景并不是多线作战，更常见的是单点突破，比如我工作就只做工作的活，生活就只管生活，那么Agent其实是存在浪费的。

因此在队列和Agent强绑定的假设排除后，我们需要直面的是并发问题：Agent需要抢任务。

如果是弱绑定呢，比如推拉模型。当Agent用完后，之前上下文的使命也告一段落了，因此拉新任务来做是很理所当然的。从本质上来说，这里变成了一个队列对应多个Agent，关键只有解决并发问题。

先避开并发这种工程问题不谈，当某一队列任务过多时，推拉模型其实会被演变成全局队列。比如现在用户正在vibecoding，有大量任务需要处理，那么之前假设要用于生活、健身的Agent都会变成CodingAgent。

因此，既然推拉模型还需要做队列之间互相监视偷任务，效果近似于全局队列，那不然直接采用全局队列。

......

告别了弱绑定，紧接着会出现致命问题：

1. 用户的任务优先级混在一起了，健身和Coding的任务可能同处一个队列，但可能Coding是比健身重要的。
2. 同一类任务之间亦有差距，比如Coding，先出方案，再写代码。这是不太好颠倒的。那如何来评定这个优先级呢，又需要Agent。而我们现在就陷入了“先有鸡还是蛋”的陷阱。我需要Agent调度Agent？

......

一切都弄混了。

我们完全可以再加一套队列逻辑，将同一类任务看成进程，这些具体的任务看成线程，也就是执行单元。这样会不会简单很多，至少同一类任务的优先级能被内部解决了（也许用Agent的工作方式）。我们将进程放入第一套队列，而不是任务，具体的操作在进程中封装。这样队列中只剩下多个不同类型的任务。[从左到右]

![](/mylovelyblog/assets/images/Agent调度策略选型/Snipaste_2026-04-08_16-22-32.png)

哈哈，这样还是将任务抽离成队列，但实际上并没有和Agent绑定了。

第一个致命问题还没有解决，健身和Coding的任务可能同处一个队列，但可能Coding是比健身重要的。我们从最初开始思考，操作系统是尽可能控制任务，而不是控制cpu，调度和执行分离。为什么要去猜测谁更重要呢，用户当前交互得多的才是重要的！因此自然想到多级反馈优先队列去解决问题，让队列根据交互做决定。

多级反馈队列规则：

1. 同级别轮转
2. 按队列优先级执行
3. 工作时先加入最高级优先级
4. 用完时间配额则下放
5. 定时提级

这样，当用户在Coding界面聊天时，需要给出的健身建议的Agent可能会在第一套队列中的第二级优先级排队。当用户在健身房撸铁询问建议时，CodingAgent也排队等着呢。

......

没有问题了吗？

可以看到，这样按照主题放入反馈队列后调度简单了许多。但我们可能会有五次并发机会，现在只有两个主题，剩下的就该浪费吗？并非如此，若反馈队列任务小于Agent并发数，我们可以通过调度，让多个Agent进入主题队列。这里我们需要维护主题队列中的优先级和Agent的关系，甚至还有抢占的逻辑，在架构层面这里不宜展开。

因此最终定型：

双队列机制：一级队列根据主题区分，二级队列根据优先级区分。

一级队列中，优先级由交互程度确定，其实现采用多级反馈调度。

二级队列中，优先级由人/Agent确定，不需要显式操作即可完成。

![](/mylovelyblog/assets/images/Agent调度策略选型/Snipaste_2026-04-08_17-18-38.png)

需要解决的工程问题：

1. 并发：多Agent干一个主题的工作，当Agent池已满，新请求如何被满足。
2. 维护：队列与Agent的关系，如何排优先级。

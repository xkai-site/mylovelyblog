这两天尝试做一个东西，让它能够自动化处理文件。主要分成四个模块：

- 文件系统
- 调度队列
- Agent
- API管理

我尝试使用git分模块开发，很快会遇到以下问题，我也给出了解决方案：

1. 四个子分支分别开发时多个终端只能进入一个分支
   方案：此处借用git另外一个机制worktree完成，这里相当于从某个分支copy了一份新文件夹，增删都很方便

2. 文档难以实时共享。当我想要加一个统一架构说明书时，从master出发需要同步给另外四个节点非常麻烦，此外同步的权限在各分支的pull手里。

   方案：

   - 使用aop，脚本做补丁操作[Maybe推荐只维护一个shared.md，然后单独使用git add shared.md等方法去写脚本]
   - 不用git，而是使用其余共享文档机制，适用于多人开发而不是多Agent。

3. 父子分支操作：

   master已更新，child需要拉取：
   child先暂存内部开发，再pull获取master更新

   child已更新，master需要合并：master通过merge合并

----

根据第三点附上更详细的多分支开发指南，可以直接给LLM喂：

核心规则：**统一使用父分支做各分支合并工作，子分支间不交互。**
child分支开发：

1. 先通过add+commit完成内部暂存与提交
2. 再使用pull拉取主分支，若冲突则解决后再使用add+commit完成主子合并提交，无冲突则可继续开发或push
3. [多人同分支开发可选]再pull拉取child远程更新,若冲突则解决后再使用add+commit完成队友合并提交
4. git push origin [child分支]

master分支开发指南：

1. 先通过add+commit完成内部暂存与提交
2. [多人同分支开发可选]再pull拉取master远程更新,若冲突解决后再使用add+commit完成队友合并提交
3. 再使用merge合并子分支，若冲突则解决后再使用add+commit完成主子合并提交，无冲突则可继续开发或push
4. git push origin [master分支]
